## 1. 项目目标与边界

本 MVP 面向“市场投放线索进入系统后，经过外呼清洗、加微承接、私域跟进，并最终转化为 MQL/SQL”的核心流程。系统不追求完整 CRM，而是验证业务抽象、状态流转、数据持久化、AI-coding 落地和演示可解释性。

本版本包含线索录入、CSV 导入去重、线索筛选、状态流转、销售分配、跟进记录、MQL/SQL 标记、漏斗统计、异常线索提醒、模拟三方回调、自然语言查询和简单角色视图。不包含真实登录、真实外呼/企微 API、复杂权限审批和生产级部署监控。

## 2. 核心业务对象

- 线索：市场投放或表单产生的潜在客户，是状态流转的主对象。
- 销售/运营负责人：承接线索的人，MQL/SQL 前必须明确负责人。
- 外呼记录/回调：外呼系统判断线索是否接通、是否有效。
- 加微状态：有效线索进入企微/SCRM 承接，记录加微成功或失败。
- 跟进记录：销售或运营对线索的沟通过程，可多次记录。
- MQL：营销认可线索，代表线索质量达到继续推进标准。
- SQL：销售认可线索，代表已有明确销售机会。
- 异常线索：有效或加微后的线索超过 48 小时未跟进，进入异常列表。
- 状态历史与操作日志：用于审计、面试追问和还原关键操作。

## 3. 核心字段设计

线索表 `lead_record` 包含：客户姓名、手机号、来源、渠道、当前状态、外呼状态、加微状态、线索等级、负责人、无效原因、备注、创建时间、分配时间、最近跟进时间、MQL 时间、SQL 时间。

负责人表 `sales_user` 包含：姓名、角色、团队。跟进记录表 `follow_up_record` 包含：线索、跟进人、方式、内容、下次跟进时间、创建时间。状态历史表 `status_history` 记录从哪个状态变到哪个状态、操作者、原因和时间。操作日志表 `operation_log` 记录新增、分配、状态变更、导入、回调等审计事件。

手机号作为 MVP 去重主键：同一手机号重复导入时不新增，只记录重复数量。

## 4. 状态流转设计

状态机如下：

```text
NEW
→ PENDING_CALL
→ CALLED_CONNECTED / CALLED_NOT_CONNECTED
→ VALID / INVALID
→ PENDING_WECHAT
→ WECHAT_ADDED / WECHAT_FAILED
→ FOLLOWING
→ MQL
→ SQL
```

关键约束：

- 无效线索不能直接标记 SQL。
- 标记 SQL 前必须先分配负责人。
- SQL 必须从 MQL 转化，避免 MQL/SQL 失去业务含义。
- 标记 MQL 前至少要有一次跟进记录。
- MQL/SQL 状态变更会自动记录对应时间。
- 每次状态变更都写入状态历史和操作日志。

## 5. MVP 页面与功能

- 线索工作台：查看线索列表，支持按状态、来源、负责人和关键词筛选。
- 新增/导入：支持表单新增线索，支持 CSV 导入并按手机号去重。
- 详情抽屉：展示线索基础信息、跟进记录、状态历史和审计日志。
- 状态变更：支持外呼、有效/无效、加微、MQL、SQL 等状态操作。
- 销售分配：主管可以把线索分配给销售。
- 漏斗看板：展示状态数量、有效率、加微率、MQL 转化率、SQL 转化率和渠道 SQL 转化。
- 异常提醒：展示超过 48 小时未跟进的有效线索。
- 自然语言查询：规则匹配回答“哪个渠道 SQL 转化率最高”等问题。
- 三方回调模拟：模拟外呼系统和企微/SCRM 状态回传。
- 角色切换：主管看全量，销售只看自己负责的线索。

## 6. 三方系统对接设想

- 投放平台：提供姓名、手机号、来源、渠道、投放计划 ID、导入时间。
- 外呼系统：通过回调返回是否接通、是否有效、无效原因、通话结果。
- 企业微信/SCRM：通过回调返回是否加微成功、外部联系人 ID、失败原因。
- 销售系统：接收 SQL 线索，后续可以生成商机、合同或跟单任务。

真实落地时需要处理回调幂等、重复回传、失败重试、字段映射、权限隔离和审计回滚。本 MVP 用模拟接口覆盖主要链路。

## 7. 关键业务指标

- 线索总数。
- 有效率 = 有效及后续阶段线索数 / 总线索数。
- 加微率 = 已加微及后续阶段线索数 / 有效线索数。
- MQL 转化率 = MQL 与 SQL 数 / 总线索数。
- SQL 转化率 = SQL 数 / 总线索数。
- 渠道 SQL 转化率 = 某渠道 SQL 数 / 某渠道线索数。
- 超时未跟进线索数。

## 8. 主动假设

- 同一手机号视为同一业务线索，重复导入不新增。
- 外呼接通且客户有兴趣时可标记为有效。
- 加微成功并完成至少一次跟进后可标记 MQL。
- MQL 后确认明确销售机会才可标记 SQL。
- 有效、已加微、跟进中、MQL 状态超过 48 小时未跟进算异常。
- 本系统采用 H2 文件库完成本地持久化，便于评审者快速启动。

## 9. 工程加固说明

- 权限边界：前端角色切换只负责演示，后端通过 `X-User-Id` 模拟当前登录用户，并根据 `sales_user.role` 强制过滤。销售用户只能读取自己负责的线索，主管可以读取全量。
- 状态机：后端采用白名单流转表，禁止跳过核心业务阶段或从 SQL 随意回退。
- 时间语义：自然语言查询中的“本周”按本周创建线索和本周 SQL 时间计算，不再使用全量数据冒充本周结果。
- 并发控制：线索表增加 `version` 乐观锁字段，用于避免并发状态变更互相覆盖。
- 导入语义：只有 CSV 导入创建的线索才写入 `imported_at`，手动新增只记录 `created_at`。
