# 市场线索到 MQL/SQL 的简化系统：业务设计文档

## 1. 背景与目标

本项目面向 DTP 数字化运营平台中的“市场线索清洗与销售流转”场景。真实业务里，市场部通过广告投放、官网表单、社群活动、合作渠道等方式获取大量线索，但这些线索并不都具备销售价值。线索进入系统后，需要经过外呼清洗、企微/SCRM 承接、销售跟进、营销认可和销售认可，最终沉淀为可继续推进的 SQL。

本 MVP 的目标不是实现完整 CRM，而是在有限时间内跑通一条可演示、可解释、可追问的主流程：

```text
新增/导入线索 -> 外呼清洗 -> 加微承接 -> 销售分配 -> 跟进 -> MQL -> SQL -> 漏斗统计/异常提醒/自然语言查询
```

核心验证点包括：

- 能否把业务流程抽象成对象、字段和状态机。
- 能否用可运行的前后端系统承载流程，而不是只画流程图。
- 能否解释 MQL、SQL、异常线索、三方回调、审计日志等业务含义。
- 能否说明哪些是 MVP 范围，哪些需要真实生产环境继续建设。

## 2. 系统边界与取舍

本版本包含以下能力：

- 线索新增、CSV 导入、手机号去重。
- 线索列表筛选、关键词搜索、角色视图。
- 外呼结果和企微/SCRM 加微结果模拟回调。
- 销售负责人分配和跟进记录。
- MQL/SQL 状态流转和时间记录。
- 状态历史、操作日志、三方回调日志。
- 漏斗看板、异常线索提醒、规则版自然语言查询。
- README、schema、API 文档、Docker Compose 和演示脚本。

本版本暂不包含：

- 真实登录注册、JWT、组织架构和复杂权限审批。
- 真实外呼系统、企微、SCRM、投放平台 API 对接。
- 生产级消息队列、任务调度、监控告警和灰度发布。
- 大数据量下的分库分表、OLAP 统计和复杂报表。

取舍原因：作业重点是业务抽象、AI-coding 落地和可解释性，因此优先保证主流程闭环、数据模型清楚、状态规则可控，而不是堆叠过重的生产基础设施。

## 3. 用户角色与典型场景

### 3.1 市场/运营

市场或运营负责导入投放线索、观察渠道质量、查看有效率和 SQL 转化率。典型问题是：

- 哪些渠道带来的线索质量更高？
- 今天导入的线索有多少重复？
- 本周哪个渠道 SQL 转化率最高？

### 3.2 外呼/企微系统

外呼系统负责电话清洗，判断是否接通、是否有效。企微/SCRM 负责加微承接，判断是否加微成功。本 MVP 不连接真实系统，而是通过回调页面模拟外部系统事件。

### 3.3 销售

销售负责承接已分配线索，记录跟进内容，并在满足条件后推动线索从已加微进入 MQL，再进入 SQL。销售视角只能看到自己负责的线索。

### 3.4 主管

主管可以查看全量线索、分配负责人、观察漏斗、处理异常线索，并追溯关键操作。主管视角用于演示管理侧看板和权限差异。

## 4. 核心业务对象

### 4.1 线索 Lead

线索是系统主对象，代表一个潜在客户机会。线索从市场进入系统后，会随着外呼、加微、跟进和销售判断不断变化状态。

关键字段：

- 客户姓名、手机号：识别客户和去重。
- 来源、渠道：用于分析不同投放来源效果。
- 当前状态：表示线索所在业务阶段。
- 外呼状态：未外呼、已接通、未接通。
- 加微状态：未开始、待加微、已加微、失败。
- 负责人：当前销售或运营承接人。
- 线索等级：A/B/C/未知，用于初步优先级判断。
- 无效原因：记录空号、无需求、重复、暂不符合等原因。
- 创建时间、分配时间、最近跟进时间、MQL 时间、SQL 时间。

### 4.2 销售用户 Sales User

销售用户用于模拟权限和负责人分配。字段包括姓名、角色、团队。角色分为 SALES 和 MANAGER。

### 4.3 跟进记录 Follow-up

一条线索可以有多次跟进记录。跟进记录用于判断线索是否被销售承接，也用于支持 MQL 前必须有跟进的业务约束。

字段包括：线索 ID、跟进人、跟进方式、跟进内容、下次跟进时间、创建时间。

### 4.4 状态历史 Status History

状态历史记录每一次状态从 A 到 B 的变化，包括操作者、原因和时间。它解决的问题是：评审或主管追问“为什么这条线索从有效变成 SQL”时，系统可以还原过程。

### 4.5 操作日志 Operation Log

操作日志记录新增、导入、分配、状态变更、跟进、回调等关键动作。与状态历史相比，操作日志更偏审计；状态历史更偏业务流转。

### 4.6 导入批次 Import Batch

记录 CSV 导入文件名、总行数、新增数、重复数、失败数和操作人，方便解释批量导入结果。

### 4.7 三方回调日志 Callback Log

记录外呼系统、企微/SCRM 的模拟回调事件，包括系统名、事件类型、回调结果、原始 payload 和接收时间。

## 5. 状态流转设计

### 5.1 状态定义

```text
NEW：新线索，刚进入系统。
PENDING_CALL：待外呼，等待电话清洗。
CALLED_CONNECTED：已接通，电话接通但还未最终判断有效性。
CALLED_NOT_CONNECTED：未接通，需要后续重拨或标记无效。
VALID：有效线索，客户身份和基础意向通过清洗。
INVALID：无效线索，空号、无需求、重复或不符合业务条件。
PENDING_WECHAT：待加微，有效线索等待企微/SCRM 承接。
WECHAT_ADDED：已加微，客户已进入私域承接。
WECHAT_FAILED：加微失败，客户拒绝或企微添加失败。
MQL：营销认可线索，具备明确需求或进一步培育价值。
SQL：销售认可线索，销售确认存在明确商机。
```

### 5.2 主流程

```text
NEW
-> PENDING_CALL
-> CALLED_CONNECTED / CALLED_NOT_CONNECTED
-> VALID / INVALID
-> PENDING_WECHAT
-> WECHAT_ADDED / WECHAT_FAILED
-> MQL
-> SQL
```

### 5.3 关键约束

- 无效线索不能直接转 SQL。
- SQL 必须从 MQL 转化，避免 SQL 变成随意打标。
- SQL 必须已有负责人，否则销售机会没有承接人。
- 已加微后可以转 MQL，但 MQL 前至少要有一次跟进记录，否则缺少销售/运营判断依据。
- MQL/SQL 自动记录时间，用于后续转化率和周期分析。
- 每次状态变化都写入状态历史和操作日志。
- 后端使用白名单状态机，禁止 NEW 直接跳到 MQL、SQL 回退到 NEW 等不合理路径。

### 5.4 为什么要区分 MQL 和 SQL

MQL 是市场或运营视角的“质量认可”：客户有效、有初步兴趣、值得继续培育。SQL 是销售视角的“商机认可”：销售确认有明确需求、预算、时机或下一步销售动作。

区分两者有三个价值：

- 衡量市场质量：如果 MQL 多但 SQL 少，说明线索兴趣可能浅或销售承接有问题。
- 衡量销售承接：如果有效线索迟迟不转 MQL/SQL，说明跟进不及时或客户培育不足。
- 支持漏斗分析：可以分清“投放问题、清洗问题、加微问题、销售转化问题”分别发生在哪个阶段。

## 6. 页面设计

### 6.1 线索工作台

展示线索列表，字段包括客户、手机号、来源、渠道、状态、负责人、最近跟进时间。支持关键词、状态、来源筛选。销售只能看自己线索，主管看全量。

### 6.2 新增与导入

新增用于单条线索录入；CSV 导入用于批量线索进入。导入按手机号去重，返回新增、重复、失败数量。

### 6.3 线索详情

详情页展示基础信息、跟进记录、状态历史和操作日志。它不是简单查看页，而是用于解释一条线索从进入到转化的完整证据链。

### 6.4 状态变更

状态变更只展示当前状态允许流转的目标状态。这样可以降低演示误操作，也体现后端状态机规则。

### 6.5 销售分配

主管可以把线索分配给张三或李四。销售角色不能分配，避免普通销售把线索任意改给别人。

### 6.6 三方回调

三方回调页面用于模拟外呼系统和企微/SCRM 的事件回传：

- 外呼回调：接通/未接通、有效/无效、备注。
- 企微回调：加微成功/失败、企微外部联系人 ID、失败原因。

页面使用搜索列表展示完整候选池，不能回传的线索置灰，原因是当前回调结果不符合状态机规则。例如 SQL 线索不能再被外呼回调改成有效。

### 6.7 漏斗看板

展示各状态数量、有效率、加微率、MQL 转化率、SQL 转化率和渠道 SQL 转化。用于回答“业务整体跑得怎么样”。

### 6.8 异常提醒

展示超过 48 小时未跟进的有效、待加微、已加微或 MQL 线索。用于主管发现销售承接风险。

### 6.9 自然语言查询

自然语言查询采用规则匹配实现，不依赖真实 LLM API。当前支持类似“本周哪个渠道 SQL 转化率最高”的问题。MVP 中重点展示 Agent/Data Query 的潜力，而不是追求复杂 NLP。

## 7. 三方系统对接设想

### 7.1 投放平台

可能字段：线索姓名、手机号、来源、渠道、广告计划 ID、创意 ID、落地页 ID、提交时间、原始表单内容。

### 7.2 外呼系统

可能回调字段：

- leadId 或 phone：用于匹配线索。
- connected：是否接通。
- valid：是否有效。
- invalidReason：无效原因。
- callDuration：通话时长。
- recordingUrl：录音地址。
- callbackTime：回调时间。

需要考虑的问题：重复回调、延迟回调、失败重试、回调签名校验、状态幂等。

### 7.3 企业微信/SCRM

可能回调字段：

- leadId 或 phone。
- externalUserId：企微外部联系人 ID。
- added：是否加微成功。
- failedReason：失败原因。
- addTime：添加时间。
- ownerUserId：企微承接员工。

需要考虑的问题：客户被多个销售添加、客户删除好友、客户转移负责人、企微与 CRM 用户映射。

### 7.4 销售/CRM 系统

当线索达到 SQL 后，可以继续创建商机、跟单任务、报价、合同或回款记录。本 MVP 到 SQL 为止，不继续实现合同和订单。

## 8. 关键指标设计

- 线索总数：观察线索池规模。
- 有效率 = 有效及后续阶段线索数 / 总线索数。
- 加微率 = 已加微及后续阶段线索数 / 有效及后续阶段线索数。
- MQL 转化率 = MQL 与 SQL 数 / 总线索数。
- SQL 转化率 = SQL 数 / 总线索数。
- 渠道 SQL 转化率 = 某渠道 SQL 数 / 某渠道线索数。
- 超时未跟进数 = 超过 48 小时未跟进的有效及后续线索数。
- 平均响应时长：从创建到首次跟进的平均时间，当前文档设计保留，MVP 未重点展示。

指标解释时需要注意：MVP 数据量较小，指标用于演示计算逻辑，不代表真实经营结论。

## 9. 权限与审计设计

本版本不做完整登录系统，但不是完全依赖前端假权限。前端角色切换只用于演示，后端通过 `X-User-Id` 模拟当前登录用户，并根据 `sales_user.role` 执行数据过滤：

- MANAGER：可以查看全量线索。
- SALES：只能查看自己负责的线索。

关键操作都会记录审计日志，包括：

- 新增线索。
- CSV 导入。
- 分配负责人。
- 状态变更。
- 新增跟进。
- 外呼/企微回调。

真实生产环境中，应使用 JWT/session 获取当前用户，而不是让前端传用户 ID。同时需要增加接口级权限、字段级权限和敏感数据脱敏。

## 10. 数据持久化与技术选型

后端采用 Spring Boot 3 + Java 21 + Maven，前端采用 Vue 3 + Vite + Element Plus。数据库采用 H2 文件库，原因是：

- 作业评审环境不一定安装 MySQL。
- H2 文件库可以本地持久化，启动成本低。
- 便于通过 zip 和 README 让评审者 10 分钟内跑起来。

为什么没有使用 MySQL：MySQL 更适合生产环境，但会增加评审者安装、账号、端口、初始化脚本等成本。MVP 阶段更重视可运行和可演示。真实落地时可以把 H2 替换为 MySQL/PostgreSQL，并配合 Flyway/Liquibase 管理 schema 版本。

## 11. 异常与边界规则

- 手机号重复：新增或导入时按手机号去重。
- 无效线索：不能进入 SQL。
- 未分配线索：不能进入 SQL。
- 未跟进线索：不能进入 MQL。
- 三方回调：只允许作用在状态机允许的阶段；不符合条件的线索在前端置灰，后端也会拒绝。
- 并发变更：线索实体使用 `@Version` 乐观锁，降低并发覆盖风险。

## 12. 后续迭代计划

第一周可以优先补齐：

- 真实登录/JWT。
- 回调签名校验和幂等键。
- 导入模板下载和导入错误明细。
- 状态回滚审批。

第二周可以继续建设：

- 企微用户映射和客户转移。
- 销售任务提醒和 SLA。
- 更完整的渠道归因和投放成本字段。
- Text-to-SQL 查询白名单。

一个月内可以扩展为：

- 可配置状态机。
- 大数据量导入队列和异步任务。
- 统计预聚合与报表缓存。
- CRM 商机、合同、回款的后续链路。

## 13. 主动假设

- 同一手机号视为同一线索主体，重复导入不新增。
- 外呼接通且客户表达基础兴趣时可标记为有效。
- 加微成功并完成至少一次跟进后，才具备进入 MQL 的依据。
- MQL 后由销售确认明确商机，才可进入 SQL。
- 有效、待加微、已加微、MQL 状态超过 48 小时未跟进算异常。
- 本 MVP 使用模拟数据，不使用任何真实客户隐私数据。
