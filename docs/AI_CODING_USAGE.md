# AI-coding 使用说明

## 1. 使用工具

本项目允许并鼓励使用 AI-coding，但我没有把 AI 输出当作最终答案直接提交，而是把 AI 作为需求拆解、代码生成、调试和文档整理的辅助工具。

使用工具包括：

- ChatGPT / Codex：用于需求拆解、数据建模、接口设计、前后端代码生成、测试用例、README 和文档初稿。
- Maven / npm / Vite：用于验证 AI 生成代码是否能真实构建。
- Git：用于版本管理和提交回溯。
- 人工复核：重点检查状态机、权限边界、自然语言查询语义、测试稳定性、文档真实性和演示可解释性。

## 2. AI 参与的具体环节

### 2.1 需求拆解

我先让 AI 根据作业要求拆出核心业务链路：

```text
新增/导入线索 -> 外呼清洗 -> 加微承接 -> 销售分配 -> 跟进 -> MQL -> SQL -> 漏斗统计/异常提醒/自然语言查询
```

AI 初步给出了模块清单，但我人工补充了状态流转约束、MQL/SQL 含义、三方回调、审计日志和面试追问场景。

### 2.2 数据建模

AI 辅助生成了以下核心对象：

- `lead_record`：线索主表。
- `sales_user`：销售/主管用户。
- `follow_up_record`：跟进记录。
- `status_history`：状态历史。
- `operation_log`：操作日志。
- `import_batch`：导入批次。
- `third_party_callback_log`：三方回调日志。

人工复核时，我重点检查每个表是否能回答面试追问，例如“谁把线索改成 SQL”“为什么变成无效”“外呼回调有没有记录”。

### 2.3 后端接口和业务规则

AI 辅助生成了 Spring Boot Controller、Service、Repository 和 DTO。接口包括线索列表、新增、状态修改、分配、跟进、CSV 导入、漏斗、异常提醒、自然语言查询和回调模拟。

人工修正重点是：

- 状态流转不能只靠前端限制。
- 角色权限不能只靠前端传参。
- MQL/SQL 必须有明确业务条件。
- 所有关键操作必须写历史和审计日志。

### 2.4 前端页面

AI 生成了 Vue + Element Plus 的初版界面，包括线索工作台、看板、异常页、查询页和回调页。之后我根据演示体验做了多轮人工调整：

- 页面从普通表格优化为更像 SaaS 工作台。
- 回调页从下拉框改成搜索列表，避免搜索不稳定。
- 状态修改弹窗默认不再自动选第一个目标状态，避免误操作。
- 分配负责人改成明确弹窗保存，避免按钮逻辑混乱。

### 2.5 测试和交付

AI 辅助生成了后端单元测试、README、schema、API 文档、演示脚本和 Docker Compose。人工重点验证：

- `mvn test` 能通过。
- `npm run build` 能通过。
- README 的本地启动和 Docker 启动说明完整。
- zip 解压后具备完整目录和文档。

## 3. 代表性 Prompt 示例

### Prompt 1：需求拆解

```text
请基于“市场线索到 MQL/SQL 的简化系统”作业要求，拆解一个 6-8 小时内可完成的 MVP。
要求包含核心对象、状态流转、接口、前端页面、加分项和交付文档。
不要做完整 CRM，要突出业务闭环和可解释性。
```

价值：帮助我把题目从“做一个系统”拆成明确模块，避免一开始就陷入 UI 或数据库细节。

### Prompt 2：状态机设计

```text
请设计线索从 NEW 到 SQL 的后端状态机，要求使用白名单流转表。
限制：INVALID 不能转 SQL；SQL 必须已有负责人；MQL 前必须已有跟进记录；
所有状态变化要写 status_history 和 operation_log。
```

价值：让 AI 先给出实现骨架，再由我人工检查哪些路径不符合业务。

### Prompt 3：自然语言查询

```text
请实现一个不依赖真实 LLM API 的规则版自然语言查询。
至少支持“本周哪个渠道 SQL 转化率最高”，注意“本周”不能用全量历史数据冒充。
```

价值：明确避免 API Key 依赖，同时保留 Agent/Data Query 的展示点。

### Prompt 4：文档和演示

```text
请根据项目功能生成 README、业务设计文档、AI-coding 使用说明和 3-5 分钟演示脚本。
文档要能支撑面试追问，不要只写功能清单。
```

价值：让 AI 给出文档初稿，再由我结合实际代码和问题修正补充。

## 4. AI 输出中的具体问题与人工修正

### 4.1 角色权限最初是“前端假安全”

问题：AI 初版只在前端做角色切换，销售看自己线索是通过前端传 `ownerId` 实现。直接调用接口时，如果传 `ownerId=null`，理论上可以看到全量数据。

修正：前端发送模拟登录头 `X-User-Id`；后端根据 `sales_user.role` 强制决定数据范围。销售用户无论前端传什么 ownerId，都只能读取自己的线索；主管才能查看全量。

### 4.2 状态流转校验太宽松

问题：AI 初版只拦截少数非法情况，例如无效线索不能 SQL、SQL 必须负责人。但 `NEW -> MQL`、`SQL -> NEW`、`WECHAT_FAILED -> MQL` 等不合理路径可能被放过。

修正：后端增加 `ALLOWED_TRANSITIONS` 白名单状态机，只允许业务文档声明的路径。同时保留 MQL 必须有跟进、SQL 必须有负责人等额外规则。

### 4.3 自然语言查询的“本周”语义不准确

问题：AI 初版识别到“渠道 + SQL + 最高”后直接用全量数据计算，没有真正处理“本周”。

修正：规则查询中识别“本周/this week”，按本周创建线索和本周 SQL 时间计算渠道转化率，避免结果和用户问题不一致。

### 4.4 测试用例硬编码 ID

问题：AI 初版测试里写死 `ownerId=1L`，依赖初始化数据顺序。换数据库或初始化顺序后测试可能失败。

修正：测试通过 `SalesUserRepository` 按姓名或角色动态查找用户，再执行分配和状态流转。

### 4.5 CSV header 识别不够健壮

问题：AI 初版只识别 `phone` 或 `手机号`，如果 CSV 表头写成“姓名,电话,来源,渠道”，第一行可能被误当成数据。

修正：扩展表头识别，支持 `phone/mobile/tel/电话/手机/姓名,电话/姓名,手机` 等常见写法。

### 4.6 手动创建线索也被标记为导入

问题：AI 初版在实体 `prePersist()` 中给所有新线索设置 `importedAt`，导致手动新增也像 CSV 导入。

修正：只在 CSV 导入逻辑里设置 `importedAt`，手动新增只记录 `createdAt`。

### 4.7 缺少并发控制

问题：AI 初版状态更新是 read-check-write。如果两个人同时修改状态，后提交的人可能覆盖前一次变更。

修正：在线索实体增加 `@Version` 乐观锁字段，schema 增加 `version`，降低并发状态覆盖风险。真实生产中还应配合错误提示和重试策略。

### 4.8 前端 `App.vue` 过重

问题：AI 初版把 5 个 Tab、多个 Dialog、全部 API 调用和状态管理都放在一个文件里，不利于扩展。

修正：拆出 `DashboardPanel`、`AnomalyPanel`、`QueryPanel` 等组件；`App.vue` 保留主流程协调。MVP 没有继续拆到很细，是为了控制复杂度。

### 4.9 三方回调页交互不自然

问题：AI 初版回调页使用下拉框，搜索体验不稳定，而且候选数据容易受当前线索列表筛选影响，看起来像只能模拟两个人。

修正：新增独立候选线索接口；前端回调页改成搜索列表，显示完整候选池。可回传线索可以点击，不符合状态机的线索置灰并标记“不可回传”。

### 4.10 构建环境兼容问题

问题：Spring Boot 默认 Maven 插件对 Maven 版本要求较高，本机 Maven 版本较低，可能导致打包失败。

修正：在 `pom.xml` 中锁定兼容版本的 Maven 插件，保证本地环境可以构建。

## 5. 人工判断与取舍

### 5.1 为什么使用 H2 而不是 MySQL

AI 曾建议使用 SQLite 或 MySQL。最终选择 H2 文件库，是因为评审环境不一定安装 MySQL，而 H2 文件库能满足本地持久化、快速启动和 zip 交付。真实生产中会迁移到 MySQL/PostgreSQL，并使用 Flyway 或 Liquibase 管理 schema。

### 5.2 为什么自然语言查询不用真实 LLM API

作业要求可以用规则、模拟 LLM、Text-to-SQL 或 Agent。考虑到 API Key、网络依赖和评审可运行性，本项目采用规则版自然语言查询。这样可以稳定演示“本周哪个渠道 SQL 转化率最高”，同时保留未来接入 LLM 的扩展空间。

### 5.3 为什么不做完整登录系统

完整登录会增加用户注册、密码加密、token 刷新、权限配置等额外复杂度。MVP 中使用 `X-User-Id` 模拟登录身份，后端仍然执行角色过滤，既能展示权限思路，又不会让登录系统喧宾夺主。

### 5.4 为什么状态机要后端兜底

前端限制只能提升用户体验，不能作为安全边界。真实系统里任何人都可能绕过前端直接调接口，所以状态流转、权限判断、SQL/MQL 约束必须放在后端。

## 6. AI 使用带来的效率提升

- 帮助快速生成项目骨架，减少重复样板代码。
- 帮助枚举对象和字段，降低遗漏基础表的风险。
- 帮助生成 README、API 文档和演示脚本初稿。
- 在报错时帮助定位 Maven、Vue、H2、状态机等问题。

但 AI 输出不能直接相信，尤其是权限、状态机、时间语义、并发、测试稳定性这类问题，需要人工复核。

## 7. 如果继续迭代，我会优先做什么

1. 用真实 JWT/session 替代 `X-User-Id`。
2. 给三方回调增加幂等键、签名校验、重复回调处理和失败重试。
3. 给状态回滚增加主管审批，避免销售误操作直接影响漏斗。
4. 将自然语言查询升级为受控 Text-to-SQL，增加表字段白名单、权限过滤、SQL 注入防护和查询超时。
5. 导入流程改成异步任务，支持大文件导入、失败明细下载和导入进度。
6. 漏斗统计改为预聚合或缓存，适配更大数据量。

## 8. 面试中可以主动说明的点

- AI 帮我提高了生成速度，但核心业务规则必须人工判断。
- 我重点修正了 AI 容易漏掉的三类问题：权限边界、状态机约束、时间语义。
- 这个 MVP 的目标是跑通业务闭环，不是替代生产 CRM。
- 真实生产还要继续补登录、幂等、审批、监控和大数据量性能优化。
